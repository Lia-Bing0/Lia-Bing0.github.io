I started as a software developer, building backend services, shipping CI/CD pipelines, and working inside regulated environments where a misconfiguration is not just a technical issue but a compliance failure. At M&T Bank, that meant PCI-DSS, multi-tier change approval, and production systems where security was never optional.

That engineering background is the lens I bring to security. I don't just flag what's wrong. I understand why it was built that way and what it takes to fix it without breaking everything else. Most security professionals learn to read code. I learned security after writing it, and that difference shows up in how I work.

Today I focus on cloud-native security: zero-trust pipeline design, OIDC-based identity federation, infrastructure-as-code security, and threat detection across Azure and AWS environments.

How I think

The question isn't "does this work?" It's "what risk does this introduce?" Misconfiguration is the vulnerability. Most cloud security failures aren't exotic exploits; they're defaults left unchallenged and access left unscoped.

This shapes how I approach every engagement, from OIDC federation in CI/CD pipelines, to Terraform IaC security gates, to Kubernetes workload identity. Every infrastructure decision is a security decision.
